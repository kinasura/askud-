1. Введение
В современной научно-исследовательской деятельности обеспечение безопасности и контроля доступа в специализированные помещения является критически важной задачей. Лаборатории часто содержат дорогостоящее оборудование, опасные вещества и конфиденциальные исследовательские данные, что требует строгого учета и контроля доступа персонала.
Существующие системы контроля доступа часто основаны на использовании пластиковых карт или биометрических идентификаторов, что может быть экономически нецелесообразно для небольших и средних предприятий. Кроме того, многие организации до сих пор используют бумажные журналы учета, что приводит к человеческим ошибкам и усложняет процесс анализа данных.
Разрабатываемая автоматизированная система контроля и учета доступа (АСКУД) предназначена для решения этих проблем путем внедрения простой, но эффективной системы идентификации на основе персональных PIN-кодов (4-8 цифр) и логин-парольной аутентификации с обязательной регистрацией как входа, так и выхода сотрудников.
Ключевые особенности системы:
- Веб-ориентированная архитектура с клиент-серверной моделью
- Двухфакторная система аутентификации (PIN-код и логин/пароль)
- Разграничение прав доступа по времени и дням недели
- Автоматическое формирование отчетов в различных форматах (CSV, Excel, PDF)
- Система уведомлений и мониторинга в реальном времени
- Адаптивный веб-интерфейс с поддержкой светлой/тёмной темы
- REST API для интеграции с внешними системами

Цель работы: разработка и внедрение автоматизированной системы контроля доступа, обеспечивающей повышенный уровень безопасности, прозрачности трудового процесса и эффективности управления доступом в лабораторные помещения.

4. Проектирование базы данных
4.1. Логическая модель данных
Система использует реляционную модель данных с шестью основными сущностями. Логическая модель представляет собой схему «сущность-связь»:
Сотрудники (1) ←→ (М) Расписание доступа (М) ←→ (1) Лаборатории
↓ ↓
(1) ←→ (М) События доступа (1) ←→ (М) Текущее присутствие
↓
(1) ←→ (М) Отчеты (связь через created_by)

Описание связей:
1. Сотрудник ↔ Расписание доступа: Один сотрудник может иметь несколько записей в расписании доступа (к разным лабораториям). Каждая запись в расписании принадлежит одному сотруднику.
2. Лаборатория ↔ Расписание доступа: Одна лаборатория может присутствовать в нескольких записях расписания доступа (для разных сотрудников). Каждая запись в расписании относится к одной лаборатории.
3. Сотрудник ↔ События доступа: Один сотрудник может иметь множество событий доступа (вход/выход). Каждое событие связано с одним сотрудником.
4. Лаборатория ↔ События доступа: Одна лаборатория может иметь множество событий доступа. Каждое событие происходит в одной лаборатории.
5. Сотрудник ↔ Текущее присутствие: Один сотрудник может находиться только в одной лаборатории одновременно (ограничение UNIQUE). Каждая запись о присутствии принадлежит одному сотруднику.
6. Лаборатория ↔ Текущее присутствие: В одной лаборатории может находиться несколько сотрудников одновременно. Каждая запись о присутствии относится к одной лаборатории.
7. Сотрудник ↔ Отчеты: Один сотрудник (администратор) может создать множество отчетов. Каждый отчет создан одним сотрудником.

4.2. Физическая модель данных
4.3. Описание таблиц
Таблица: employees (Сотрудники)
PK id: INTEGER (AUTOINCREMENT)
  login: VARCHAR(100) NOT NULL UNIQUE
  password: VARCHAR(100) NOT NULL
  pin_code: VARCHAR(20) NOT NULL UNIQUE
  full_name: TEXT NOT NULL
  department: VARCHAR(100)
  position: VARCHAR(100)
  phone: VARCHAR(20)
  email: VARCHAR(100)
  is_active: BOOLEAN DEFAULT TRUE
  user_type: VARCHAR(20) DEFAULT 'employee'
  created_at: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
Таблица: laboratories (Лаборатории)
PK id: INTEGER (AUTOINCREMENT)
  name: VARCHAR(100) NOT NULL
  code: VARCHAR(20) NOT NULL UNIQUE
  location: VARCHAR(200)
  description: TEXT
  capacity: INTEGER
  is_active: BOOLEAN DEFAULT TRUE
  created_at: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
Таблица: access_schedules (Расписание доступа)
PK id: INTEGER (AUTOINCREMENT)
FK employee_id: INTEGER → employees.id
FK laboratory_id: INTEGER → laboratories.id
  days_of_week: TEXT
  time_start: TIME
  time_end: TIME
UNIQUE(employee_id, laboratory_id)
Таблица: access_events (События доступа)
PK id: INTEGER (AUTOINCREMENT)
FK employee_id: INTEGER → employees.id
FK laboratory_id: INTEGER → laboratories.id
  event_type: VARCHAR(20) NOT NULL
  event_time: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  success: BOOLEAN NOT NULL
  reason: TEXT
  method: VARCHAR(20) DEFAULT 'pin'
Таблица: current_presence (Текущее присутствие)
PK id: INTEGER (AUTOINCREMENT)
FK employee_id: INTEGER → employees.id UNIQUE
FK laboratory_id: INTEGER → laboratories.id
  entry_time: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  expected_exit_time: TIMESTAMP
Таблица: reports (Отчеты)
PK id: INTEGER (AUTOINCREMENT)
  name: TEXT NOT NULL
  report_type: TEXT NOT NULL
  period_start: DATE
  period_end: DATE
  generated_at: TIMESTAMP DEFAULT CURRENT_TIMESTAMP
  file_path: TEXT
FK created_by: INTEGER → employees.id
Связи (кардинальность):
employees (1) → access_schedules (*) - Один сотрудник имеет множество расписаний
laboratories (1) → access_schedules (*) - Одна лаборатория имеет множество расписаний
employees (1) → access_events (*) - Один сотрудник создает множество событий
laboratories (1) → access_events (*) - В одной лаборатории происходит множество событий
employees (1) → current_presence (0..1) - Сотрудник может находиться в 0 или 1 лаборатории
laboratories (1) → current_presence (*) - В лаборатории может находиться множество сотрудников
employees (1) → reports (*) - Один сотрудник создает множество отчетов

4.4. Индексы
Для обеспечения высокой производительности запросов в базе данных созданы следующие индексы:
1. employees.login - уникальный индекс для ускорения поиска при аутентификации.
2. employees.pin_code - уникальный индекс для быстрой проверки PIN-кода в терминале.
3. access_events.event_time - индекс для эффективной фильтрации событий по дате при формировании отчетов.
4. access_events.employee_id, access_events.event_time - составной индекс для быстрого получения истории посещений конкретного сотрудника.
5. current_presence.employee_id - индекс для мгновенной проверки, находится ли сотрудник в лаборатории.
6. access_schedules.employee_id, access_schedules.laboratory_id - индекс для ускорения проверки прав доступа.

4.5. Триггеры и ограничения целостности
1. Автоматическое обновление текущего присутствия: При успешном событии 'entry' в `access_events` запись добавляется в `current_presence`. При успешном событии 'exit' запись удаляется из `current_presence`. (Реализовано на уровне бизнес-логики в `verify_access` функции).
2. Проверка уникальности PIN-кода: Обеспечивается ограничением UNIQUE на поле `pin_code` в таблице `employees`.
3. Проверка уникальности логина: Обеспечивается ограничением UNIQUE на поле `login`.
4. Каскадное удаление: При удалении сотрудника автоматически удаляются связанные записи в `access_schedules` и `current_presence`. При удалении лаборатории удаляются связанные записи в `access_schedules`.

4.6. Нормализация базы данных
База данных соответствует третьей нормальной форме (3NF):
- Все неключевые атрибуты зависят только от первичного ключа.
- Отсутствуют транзитивные зависимости.
- Отсутствуют повторяющиеся группы данных.

4.7. Примеры ключевых SQL-запросов
Запрос для проверки доступа сотрудника:
SELECT days_of_week, time_start, time_end 
FROM access_schedules 
WHERE employee_id = ? AND laboratory_id = ?;
Запрос для получения последних событий (для дашборда администратора):
SELECT ae.event_time, e.full_name, l.name as laboratory, 
       ae.event_type, ae.success, ae.reason
FROM access_events ae
JOIN employees e ON ae.employee_id = e.id
JOIN laboratories l ON ae.laboratory_id = l.id
ORDER BY ae.event_time DESC
LIMIT 20;
Запрос для формирования отчета за период:
SELECT ae.event_time, e.full_name, e.department, 
       l.name as laboratory, ae.event_type, ae.success, ae.reason
FROM access_events ae
JOIN employees e ON ae.employee_id = e.id
JOIN laboratories l ON ae.laboratory_id = l.id
WHERE DATE(ae.event_time) BETWEEN ? AND ?
ORDER BY ae.event_time DESC;
Запрос для получения списка сотрудников в конкретной лаборатории:
SELECT e.full_name, e.department, e.position, cp.entry_time
FROM current_presence cp
JOIN employees e ON cp.employee_id = e.id
WHERE cp.laboratory_id = ?;

7.1. Архитектурный подход и технологии
Система реализована с использованием трехзвенной архитектуры (3-tier architecture), которая обеспечивает четкое разделение ответственности между компонентами:
┌─────────────┐ HTTP/HTTPS ┌────────────┐ SQL/ORM ┌───────────────┐
│ Presentation │◄─────────►│Business│ ◄────────►│ Data │
│ Layer │ │ Logic │ │ Layer │
│ (Веб-интерфейс)│ │ Layer │ │ (База данных) │
└─────────────┘ └─────────────┘ └─────────────┘

Технологический стек:
- Фронтенд (Presentation Layer): HTML5, CSS3, JavaScript, Bootstrap 5, Chart.js
- Бэкенд (Business Logic Layer): Python 3.8+, Flask 2.0+, SQLAlchemy (опционально)
- База данных (Data Layer): SQLite 3.35+ с миграцией на PostgreSQL для production
- API: RESTful API с JSON форматом данных
- Дополнительные библиотеки: pandas (анализ данных), FPDF/pdfkit (генерация PDF), openpyxl (работа с Excel)

8.1. Методология тестирования
Тестирование системы проводилось в соответствии с ГОСТ Р 56546-2015 "Системы автоматизированные. Методы испытаний" и международным стандартом ISO/IEC/IEEE 29119. Применялся комбинированный подход, включающий следующие виды тестирования:

2. Интеграционное тестирование (Integration Testing): тестирование взаимодействия компонентов
3. Системное тестирование (System Testing): тестирование системы в целом
4. Нагрузочное тестирование (Load Testing): тестирование под нагрузкой
5. Тестирование безопасности (Security Testing): проверка защищенности системы
6. Пользовательское тестирование (User Acceptance Testing): тестирование с участием конечных пользователей

- Подготовка системы к внедрению в производственную среду

- Функциональная полнота: 100% реализация требований ТЗ
- Надежность: время наработки на отказ не менее 720 часов
- Производительность: время отклика не более 2 секунд при 50 одновременных пользователей

10.6. Расчет показателей эффективности
10.6.1. Чистая приведенная стоимость (NPV):
NPV = ∑(CF_t / (1 + r)^t) - I
Где:
CF_t = денежный поток в году t
r = ставка дисконтирования (15%)
I = первоначальные инвестиции (856 000 руб.)
t = год (1-5)
Год 1: 2 220 000 - 442 000 = 1 778 000
Год 2: 2 220 000 - 442 000 = 1 778 000
Год 3: 2 220 000 - 442 000 = 1 778 000
Год 4: 2 220 000 - 442 000 = 1 778 000
Год 5: 2 220 000 - 442 000 = 1 778 000
NPV = 1 778 000/1.15 + 1 778 000/1.15² + 1 778 000/1.15³ + 1 778 000/1.15⁴ + 1 778 000/1.15⁵ - 856 000
NPV = 1 546 087 + 1 344 423 + 1 169 063 + 1 016 576 + 884 026 - 856 000
NPV = 5 960 175 - 856 000 = 5 104 175 руб.

10.6.2. Внутренняя норма доходности (IRR):

10.6.3. Срок окупаемости (PP):
Инвестиции: 856 000 руб.
Годовой денежный поток: 1 778 000 руб.
PP = 856 000 / 1 778 000 = 0.48 года ≈ 6 месяцев

Система АСКУД обладает значительным потенциалом для развития в следующих направлениях:
1. Функциональное расширение: Добавление новых модулей и возможностей
2. Технологическая модернизация: Переход на современные технологии и архитектуры
3. Интеграционная платформа: Становление центральным элементом корпоративной системы безопасности
4. Масштабирование: Поддержка распределенных структур и увеличение числа пользователей

11.2. Краткосрочные перспективы (1-2 года)
11.2.1. Улучшение безопасности:
1. Двухфакторная аутентификация:
   - Поддержка TOTP (Time-based One-Time Password)
   - SMS-коды для критических операций
   - Аутентификация через мобильное приложение
2. Биометрические методы идентификации:
   - Распознавание лиц через камеры терминалов
   - Сканеры отпечатков пальцев
   - Голосовая идентификация
3. Расширенное шифрование данных:
   - Шифрование паролей с использованием bcrypt/argon2
   - TLS 1.3 для всех соединений
   - Шифрование данных в БД на уровне поля

11.2.2. Расширение функциональности:
1. Интеграция с календарями:
   - Синхронизация с Google Calendar/Outlook
   - Автоматическое планирование доступа по мероприятиям
   - Резервирование лабораторий
2. Мобильное приложение:
   - Уведомления о событиях в реальном времени
   - Удаленный контроль доступа
   - QR-коды для гостевого доступа
3. Расширенная аналитика:
   - Машинное обучение для выявления аномалий
   - Прогнозирование загрузки лабораторий
   - Анализ паттернов поведения

11.2.3. Улучшение пользовательского опыта:
1. Голосовой помощник:
   - Голосовые команды для терминалов
   - Голосовые отчеты для администраторов
   - Озвучивание инструкций
2. Универсальный доступ:
   - Поддержка screen readers для слабовидящих
   - Управление жестами для терминалов
   - Многоязычный интерфейс

11.3. Среднесрочные перспективы (3-5 лет)
11.3.1. Технологическая миграция:
1. Переход на микросервисную архитектуру:
   - Выделение сервисов аутентификации, контроля доступа, отчетности
   - Контейнеризация с использованием Docker/Kubernetes
   - Service mesh для управления трафиком
2. Миграция базы данных:
   - Переход с SQLite на PostgreSQL для production
   - Репликация данных для отказоустойчивости
   - Горизонтальное масштабирование
3. Облачная инфраструктура:
   - Размещение в приватном/гибридном облаке
   - Использование managed services
   - Автоматическое масштабирование

11.3.2. Интеграционная платформа:
1. Интеграция с корпоративными системами:
   - Active Directory/LDAP для централизованной аутентификации
   - 1C для учета рабочего времени
   - ERP системы для управления ресурсами
2. Интеграция с системами безопасности:
   - Видеонаблюдение (распознавание лиц, трекинг)
   - Пожарная и охранная сигнализация
   - Системы контроля периметра
3. Интеграция с IoT устройствами:
   - Датчики присутствия для автоматического учета
   - Контроль климатического оборудования
   - Управление энергопотреблением

11.3.3. Расширение области применения:
1. Поддержка новых типов помещений:
   - Серверные комнаты и ЦОДы
   - Складские помещения с опасными материалами
   - Чистые комнаты и лаборатории повышенной безопасности
2. Гостевой доступ:
   - Временные учетные записи для посетителей
   - Электронные пропуска с ограниченным сроком действия
   - Сопровождение посетителей
3. Управление активами:
   - Учет оборудования в лабораториях
   - Контроль выдачи инструментов и приборов
   - Инвентаризация в автоматическом режиме

11.4. Долгосрочные перспективы (5+ лет)
11.4.1. Искусственный интеллект и машинное обучение:
1. Предиктивная аналитика:
   - Прогнозирование попыток несанкционированного доступа
   - Оптимизация расписания доступа на основе паттернов
   - Выявление аномалий в реальном времени
2. Адаптивная безопасность:
   - Динамическое изменение правил доступа на основе угроз
   - Автоматическая реакция на инциденты безопасности
   - Самообучающаяся система обнаружения вторжений
3. Персонализированный опыт:
   - Адаптация интерфейса под индивидуальные предпочтения
   - Рекомендации по оптимальному времени посещения
   - Персональные уведомления и напоминания

11.4.2. Блокчейн технологии:
1. Неизменяемый журнал событий:
   - Хранение записей доступа в блокчейне
   - Гарантированная целостность данных
   - Аудит без возможности фальсификации
2. Децентрализованная идентификация:
   - Self-sovereign identity для сотрудников
   - Межорганизационное доверие
   - Упрощенный гостевой доступ
3. Смарт-контракты для управления доступом:
   - Автоматическое применение правил
   - Условный доступ на основе внешних данных
   - Многостороннее подтверждение доступа

11.4.3. Дополненная и виртуальная реальность:
1. AR-интерфейс для терминалов:
   - Проекция интерфейса на любую поверхность
   - Голографические подсказки
   - Бесконтактное управление
2. VR-тренажеры для обучения:
   - Виртуальные сценарии чрезвычайных ситуаций
   - Обучение правилам безопасности
   - Симуляция работы с системой
3. Цифровые двойники лабораторий:
   - Визуализация текущего состояния помещений
   - Симуляция различных сценариев
   - Оптимизация планирования использования

12.Заключение
12.1. Итоги выполненной работы
В рамках курсового проекта была разработана и реализована автоматизированная система контроля и учета доступа (АСКУД) для лабораторных помещений. Основные результаты работы:
1. Полностью функционирующая система, включающая:
   - Веб-интерфейс для различных категорий пользователей
   - Терминалы доступа с поддержкой PIN-кодов
   - Административную панель для управления системой
   - Механизмы формирования отчетов и аналитики
2. Проектная документация, соответствующая требованиям ГОСТ:
   - Техническое задание
   - Описание архитектуры и компонентов
   - Руководство пользователя
   - Результаты тестирования
3. Экономическое обоснование, подтверждающее эффективность внедрения системы

12.2. Достигнутые цели
Система успешно решает поставленные задачи:
1. Контроль доступа: Обеспечивает строгое разграничение доступа на основе персональных прав и расписания
2. Учет рабочего времени: Автоматически фиксирует время прихода и ухода сотрудников
3. Мониторинг безопасности: Выявляет и документирует попытки несанкционированного доступа
4. Формирование отчетности: Генерирует отчеты в различных форматах для анализа и документирования
5. Управление присутствием: Отслеживает текущее нахождение сотрудников в лабораториях

12.3. Ключевые особенности системы
1. Экономическая эффективность: Использование открытых технологий и стандартного оборудования снижает стоимость внедрения
2. Простота использования: Интуитивно понятный интерфейс для всех категорий пользователей
3. Гибкость и масштабируемость: Возможность адаптации под различные организации и масштабирования при росте
4. Надежность: Архитектура, обеспечивающая высокую доступность и отказоустойчивость
5. Безопасность: Многоуровневая система защиты данных и контроля доступа

12.4. Технические инновации
В системе реализованы следующие технические решения:
1. Автоматическое определение действия (вход/выход) на основе текущего состояния сотрудника
2. Поддержка сложных расписаний доступа с учетом дней недели и временных интервалов
3. Многоформатная система отчетности с экспортом в CSV, Excel и PDF
4. REST API для интеграции с внешними системами
5. Адаптивный веб-интерфейс с поддержкой светлой и темной темы

12.5. Практическая значимость
Разработанная система имеет высокую практическую значимость:
1. Для образовательных учреждений: Может использоваться в учебных лабораториях ВУЗов и колледжей
2. Для научно-исследовательских организаций: Обеспечивает безопасность и контроль доступа в специализированные помещения
3. Для промышленных предприятий: Подходит для контроля доступа в производственные лаборатории и чистые комнаты
4. Для медицинских учреждений: Может быть адаптирована для контроля доступа в лаборатории и хранилища

12.6. Перспективы внедрения
Система готова к внедрению в производственную среду и может быть развернута в течение 2-4 недель. Рекомендуется поэтапное внедрение:
1. Пилотный проект: Внедрение в одной лаборатории для отработки процедур

Для повышения эффективности системы рекомендуется:
1. Внедрить хэширование паролей перед промышленной эксплуатацией
